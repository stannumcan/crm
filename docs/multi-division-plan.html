1. Context & Decision

The CRM currently serves one division — Winhoop (Japan). A second division, Stannum Can (Canada), needs to use the same system with the same operational workflows but fully separate customer data.

Three approaches were evaluated:

Approach Pros Cons Verdict
Two separate repositories Complete isolation Every bug fix = two PRs; codebases will diverge; double maintenance Rejected
One repo, two Supabase projects Database-level isolation Env-var switching; shared data (molds) needs syncing; no cross-division reporting Rejected
One repo, one database, multi-tenant Single codebase; shared resources; RLS isolation; cross-division admin Requires careful RLS design Selected

  • Customer currency: JPY
  • Factory currency: CNY
  • Wire currency: CNY
  • WO prefix: JP (e.g., JP260045)
  • Status: Live, all existing data

Stannum Can — Canada

  • Region code: CA
  • Customer currency: USD (sometimes CAD)
  • Factory currency: USD and CNY
  • Wire currency: USD
  • WO prefix: CA (all WOs start with CA — no prior prefix existed)
  • Status: New, starting clean

Confirmed Shared Characteristics

  • Same factory — separate monthly statements per division, same format
  • Same quotation pipeline — factory sheet → cost calc → DDP → customer quote
  • Completely separate customers — no customer buys from both divisions
  • DingTalk notifications for both divisions
  • Shared mold catalog — same products available to both

3. Architecture: Single Repo, Single Database

Core principle Because operations are identical and customers are fully separate, the division is purely a filter column on existing tables. No structural schema differences, no conditional workflows, no per-division code paths.

The isolation strategy has three layers:

  1. Database layer — Row-Level Security (RLS) policies filter every query by the user's division. Even if the frontend has a bug, the database won't return data from the wrong division.
  2. API layer — Server-side routes read the user's division from their session and pass it to queries. Defense-in-depth alongside RLS.
  3. Frontend layer — UI only displays data for the active division. Admin users get a division switcher.

4. Currency Architecture

This is the main structural difference between the two divisions.

Factory billing Always CNY USD or CNY (per statement)
Wire to factory CNY USD
FX handling Bank rate on wire date Factory-provided rate for CNY → USD conversion
Design implication All financial tables need a currency field per record rather than assuming a fixed currency per division/track. Stannum Can is multi-currency on both the customer side (USD/CAD) and factory side (USD/CNY). The factory-provided exchange rate must also be stored as a field on CNY-denominated factory statements.

7. Row-Level Security

Current RLS enforces page-level permissions (can this user view quotes?). Division adds a second layer: can this user see this division's data?

New RLS helper functions

  • user_can_access_division(target_division_id uuid) — returns true if the current user has a matching row in user_divisions, or is a super-admin.
  • user_is_super_admin() — returns true if current user has is_super_admin = true.

Policy pattern

For each division-scoped table:

  • SELECT: existing page permission check AND user_can_access_division(division_id)
  • INSERT: same — user can only insert rows for a division they belong to
  • UPDATE / DELETE: same pattern

Because division_id is denormalized onto every scoped table, RLS policies don't need joins — single-column comparison only. Performance stays fast as data grows.

Why RLS matters here RLS is enforced at the database level, below the application. Even if an API route has a bug or someone crafts a direct request, the database will not return rows from the wrong division. This is the strongest guarantee against data contamination.

Admin override

Super-admin users (Wilfred) need to query across divisions for the combined dashboard. The user_can_access_division() function returns true for all divisions if is_super_admin = true. This also covers cross-division users like accounting — although they'll normally only have two divisions in user_divisions rather than the super-admin flag.

8. Authentication & User Context

Current flow

User logs in → session stored in cookies → user_profiles looked up for permissions → no division concept.

Updated flow

  1. User logs in → session stored in cookies
  2. user_profiles looked up → includes active_division_id, is_super_admin
  3. user_divisions looked up → returns list of divisions this user can access
  4. Division context loaded into a React context provider (alongside existing permissions context)
  5. All API routes read the user's accessible divisions from session
  6. RLS automatically enforces at database level via user_can_access_division()

10. Impact on Planned Financial Module

This is the highest-complexity area The financial module (not yet built) must be designed multi-currency from the start. Retrofitting currency support later would be painful.

Existing design (Winhoop-only)

Two-track model designed around fixed currencies:

  • Track 1: Customer → Winhoop (JPY)
  • Track 2: Winhoop → Factory (CNY)

Updated design (multi-division)

Same two-track model, but currency is per-record, not per-track:

Table Currency Field Winhoop Values Stannum Can Values
customer_pos currency JPY USD or CAD
customer_invoices currency JPY USD or CAD
customer_payments currency JPY USD or CAD
factory_pos currency CNY USD or CNY
factory_statements currency CNY USD or CNY
wire_transfers currency CNY USD

Additional fields for Stannum Can

  • factory_statements.fx_rate — factory-provided USD/CNY exchange rate (only when statement is in CNY)
  • factory_statements.fx_amount_usd — converted USD amount using factory's rate
  • All monetary amount fields should be numeric with the currency field indicating the denomination

12. Frontend / UI Changes

Division context provider

New React context (alongside existing PermissionsProvider) that provides:

  • accessibleDivisions — array of divisions this user can access (from user_divisions)
  • activeDivision — currently-selected division, or "all" for super-admin combined view
  • isSuperAdmin — flag controlling combined dashboard access
  • switchDivision(id | "all") — available when user has multiple divisions

Visual changes per user type

User TypeHeaderDashboardLists & Forms
Single-division users Division name shown, no switcher Their division's data only Auto-scoped by RLS
Cross-division users (accounting) Division switcher dropdown Active division's data Auto-scoped by active division
Super-admin (Wilfred) Division dropdown with "All Divisions" option (default) Combined view by default — merges JP + CA data with division badges on each row Lists show rows from all divisions with a division-badge column; can filter to one division if desired

Combined dashboard design (super-admin)

  • Default view is combined — no extra clicks needed
  • Each row in lists (work orders, quotes, etc.) has a small division badge (JP blue, CA red)
  • Dashboard widgets show merged counts with a breakdown tooltip (e.g., "Active WOs: 47 total — 32 JP, 15 CA")
  • Currency totals displayed per-currency (cannot merge JPY and USD into one number)
  • Quick filter in header to scope to single division when needed

General UI rules

  • All list views: Automatically filtered by RLS — no frontend filtering needed.
  • Forms: Currency fields default to the active division's default currency but allow override where applicable (Stannum Can customer quotes can be USD or CAD).
  • WO creation: Region/prefix auto-set from active division (JP or CA). Super-admin picks which division when creating.

No structural UI changes needed

Since both divisions use the same pipeline, all pages, forms, and workflows remain identical. The only differences are data filtering, currency display, and the division badge on list rows for admin.

14. Build Rules & Constraints

Do not touch the live database All schema changes must be built on a Supabase preview branch. Only merge to production after explicit approval.
  • Planning mode only until explicitly told to start building
  • Preview branch required — create before any migration or schema change
  • Backfill existing data as Winhoop/JP — no data loss
  • No breaking changes to existing functionality during migration
  • Financial tables must include currency field from day one — do not hardcode currencies
